## Critical — Fix First

### 1. SonarCloud isn't actually running

`sonar-project.properties` still references `Sc0red_template-typescript-backend-repo` — leftover from the repo template. The project returns 404 on SonarCloud, so static analysis has been flowing nowhere.

**Fix:** create the Clawndom project on SonarCloud and update both `projectKey` and `projectName` in `sonar-project.properties`. Zero code changes; you'll inherit a SonarCloud backlog immediately.

### 2. Coverage thresholds are failing silently

The coverage gate in `vitest.config.ts` requires 95% lines/functions/statements + 94% branches. Current reality:

- Lines: 89.9%
- Functions: 91.66%
- Statements: 89.9%
- Branches: 89.21%

`make check-all` should be blocking commits on this but isn't in practice — either the gate isn't running in CI, or recent commits bypassed it. **The "95%+ test coverage" rule in CLAUDE.md has been aspirational, not enforced.**

**Fix:** either raise the bar back to stated thresholds and write the missing specs, or lower the stated threshold to match reality (~90%). Don't let the gap stay — the rule's credibility erodes.

### 3. Connection/queue leak in failed-job handler

`src/services/worker.service.ts:335-336` creates a **new** `IORedis` connection and BullMQ `Queue` on **every failed job**:

```ts
const retryConn = new IORedis(redisUrl, { maxRetriesPerRequest: null });
const queue = new Queue(buildQueueName(provider.name), { connection: retryConn });
```

Under a bad-job storm (e.g. a downstream runner returning errors), this opens fresh Redis connections as fast as jobs fail. The `.finally()` block does close them, but with `.catch(() => {})` swallowing errors, a single failure here leaks the connection silently. Fail-fast rule is violated.

**Fix:** reuse the worker's existing `connection` for re-enqueue. Move the requeue Queue outside the failed handler and close it in worker shutdown.

### 6. `as` casting — rule says "fix the types"

10+ violations, concentrated in two files:

- `src/lib/logging/adapters/lambda-adapter.ts` — 4 casts of Lambda event-shape properties (`requestContext`, `authorizer`, `claims`, `headers`). These are from AWS Lambda event shapes — legitimately `unknown`. The right fix is a Zod schema for `APIGatewayEvent` and `.parse()` at the entrypoint, then trust the type internally.
- `src/runners/claude-cli.runner.ts` — 3 casts parsing streamed JSON events. Use Zod to define the event envelope and parse at read boundary.
- `src/config.ts:101` — `JSON.parse(raw) as ProviderConfig[]` — no runtime validation on what's effectively external config. Replace with `ProviderConfigSchema.array().parse(JSON.parse(raw))`. Critical for a config parser.
- `src/controllers/webhook.controller.ts:57` — `request.body as Buffer`. If the raw-body middleware isn't configured for a route, this cast silently turns something else into Buffer-typed code. Add a runtime guard or document the middleware contract in a comment.

## Opportunities — Worth Considering

### 8. tsconfig tightening

Two settings not enabled that would catch real bugs given Clawndom's heavy use of dynamic payloads:

- `noPropertyAccessFromIndexSignature: true` — forces `record['foo']` over `record.foo` on index-signature types. Would eliminate the `as Record<string, unknown>` pattern (#6) because access would be explicit bracket notation; you'd then drive the types through Zod narrowing instead of casting.
- `exactOptionalPropertyTypes: true` — distinguishes `{ foo?: string }` from `{ foo: string | undefined }`. Mostly catches subtle bugs where you pass `undefined` to an optional field.

Both are disruptive to enable retroactively (expect fallout), but that fallout *is* the audit.

### 9. Webhook controller defensive-logic drift

The webhook controller mixes typeof-guards (`typeof signatureHeader !== 'string'`) with business logic. Per the "Validate at boundaries only" rule, this guard logic should live in a Zod schema in the validation middleware, leaving the controller to trust its typed inputs.

### 10. The 118-files headline number

5,190 LOC across 118 files averages 44 LOC per file — a sign that the layering is working. The outliers (worker.service, webhook.controller) are where complexity clumps, and those are exactly what finding #4 and #5 surface.

## What's Working

- `strict: true` + `noUncheckedIndexedAccess: true` + `noImplicitReturns: true` + `noImplicitOverride: true` — strong foundation
- Zero `any`, zero `@ts-ignore`, zero non-null assertions in `src/` — the big-ticket TS anti-patterns aren't creeping in
- Layered architecture (`routes → controllers → services → lib`) is respected in practice, not just documented
- Runner abstraction is a clean strategy pattern; adding a new runner means one file in `src/runners/`
- Secrets providers follow the same pattern
- Structured logging via pino, with request context plumbing
